# ADR-0003: 커밋된 openapi.json을 계약 권위로 삼는 code-first 생성·승격 체제 채택

## 상태

승인됨 (2026-07-15). 승인 전제였던 springdoc 빌드 타임 스펙 추출
스파이크는 contract-gate 워크플로 구현·CI 실증(PR #35)으로 충족.

## 맥락

1막은 백엔드(maintainer 단독)와 프론트엔드(Claude Code, 전용 계정)의
병렬 개발이며, 프론트는 백엔드 코드를 읽지 않고 `openapi.json`만 보고
작업한다. 계약이 사실상 유일한 협업 채널이므로 다음 세 가지가 요구된다.

- **변경 통제**: 백엔드가 계약을 일방적으로 바꿀 수 없어야 한다.
  계약 변경은 눈에 보이는 절차(PR + 검증)를 통과해야 한다.
- **자체 검증 가능성**: 프론트는 백엔드 인프라 없이, 커밋된
  `openapi.json`만으로 자기 구현의 올바름을 검증(목 서버·스키마 검증)할
  수 있어야 한다. 로컬에 백엔드 스택을 깔지 않는다.
- **역방향 피드백**: 프론트가 화면 구현에 필요한 데이터가 계약에 없을 때
  이슈/코멘트로 변경을 제안할 수 있어야 한다. 단, 계약 파일의 쓰기
  권한은 백엔드에만 있다.

프론트가 AI라는 점은 질문 루프(모호하면 maintainer에게 질문)로 사람
개발자와의 차이가 축소되지만, *그럴듯하게 틀린* 계약(nullable 누락,
에러 스키마 부재)은 질문 없이 그럴듯하게 틀린 구현으로 이어진다.
계약 품질의 하한선이 사람 프론트 대비 높다.

도구 사실관계: springdoc-openapi 3.x가 Spring Boot 4를 공식 지원함을
확인(2026-07-13). 반면 openapi-generator의 Spring 서버 스텁 생성은
Boot 4 + Jackson 3 대응이 뒤처질 가능성이 있어 별도 검증이 필요한
상태였다.

## 결정

계약 저작은 code-first로 한다 — springdoc으로 백엔드 코드에서 스펙을
생성한다. 단, 커밋된 `openapi.json`을 계약의 유일한 권위로 삼고,
생성 스펙의 커밋 반영은 자동 발행이 아니라 maintainer의 의도적
승격(promotion)으로 한다. PR 시점에 CI가 생성 스펙과 커밋된 계약을
diff하여 불일치면 빌드를 실패시킨다. CI 통과는 머지의 필요조건이며,
`openapi.json`에 diff가 발생한 PR은 셀프머지 티어라도 공식 리뷰
티어로 승격한다. 첫 계약은 첫 도메인의 컨트롤러 스텁에서 생성·승격하여
프론트 착수를 연다.

## 결과

- **얻는 것**:
  - 계약 변경이 무의식적으로 흘러가지 않는다. 코드 리팩토링으로 계약이
    조용히 바뀌는 실패 모드(code-first 자동 발행)와, 수동 전사를 잊는
    실패 모드(게이트 없는 승격)를 모두 CI 빨간불로 전환한다.
  - 프론트는 커밋된 계약만 신뢰하면 된다. 백엔드의 미완성 상태가
    프론트에 새지 않는다.
  - openapi-generator의 Boot 4 호환성 리스크를 회피한다. 서버 코드는
    전량 수작업이므로 1막 원칙(백엔드 AI 코드 작성 없음)과도 정합.
- **잃는 것 / 감수하는 리스크**:
  - 계약 품질 작업이 Java 어노테이션 노동으로 돌아온다. springdoc이
    자동 추출하는 것은 해피패스 위주이며, 에러 응답 스키마와 예시는
    어노테이션을 의도적으로 달아야 계약에 나타난다.
  - API 설계 사고가 계약 언어가 아니라 Java에서 일어난다. "계약 작성
    연습"이 아니라 "계약 변경 관리 연습"이 된다.
  - 첫 계약이 첫 컨트롤러 스텁에 종속되어, 순수 contract-first 대비
    프론트 착수가 늦다.
- **잃는 것을 감수할 수 있는 근거**:
  - 이 프로젝트의 학습 무게중심은 설계 저작이 아니라 변경에 용이한
    상태를 만드는 것(변경 관리)에 있다고 판단했다 (2026-07-13 세션).
  - 에러 응답 상세화는 어노테이션 비용이 들지만, 그 자체가 에러 설계
    연습이라는 학습 목표와 일치한다. 비용을 상수로 누르기 위해 에러
    응답 스키마 통일(RFC 7807 / Spring `ProblemDetail`)을 별도
    검토한다(이슈 코멘트감, 첫 컨트롤러 스텁 전 확정).
  - 착수 지연은 첫 스텁 1개 분량으로 유한하다.

## 고려한 대안

- **순수 contract-first (계약 손 저작)**: 프론트가 즉시 착수 가능하고
  API 설계 사고를 계약 언어에서 하게 된다. 기각 — JSON/YAML 수작업
  마찰이 크고, 학습 무게중심이 저작이 아닌 변경 관리라는 판단과
  어긋나며, 착수 지연은 스텁 1개로 해소 가능한 수준.
- **순수 code-first (자동 발행)**: 드리프트가 원천 불가능하고 운영
  비용이 최소. 기각 — 계약 변경이 어떤 의식적 절차도 통과하지 않아,
  백엔드 리팩토링이 프론트를 조용히 깨뜨린다. 변경 통제 요구와 정면
  충돌.
- **게이트 없는 수동 승격 (생성 후 손으로 옮김, CI 검증 없음)**: 승격
  행위가 체크포인트 역할. 기각 — 옮기는 것을 잊는 실패 모드가
  침묵한다. 컴파일·테스트 모두 통과한 채 프론트만 낡은 계약을 보게
  된다.
- **openapi-generator 서버 스텁 생성**: contract-first의 통상 구현.
  기각 — Boot 4 + Jackson 3 대응 검증 부담이 있고, 생성 코드 사용이
  "백엔드 전량 수작업" 원칙과 긴장 관계.
