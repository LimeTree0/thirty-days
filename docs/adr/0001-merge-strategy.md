# ADR-0001: PR 머지 방식으로 merge commit 채택

## 상태

승인됨 (2026-07-11)

## 맥락

main은 Ruleset으로 보호되어 모든 변경이 PR을 거치므로, PR 머지 방식이
곧 main 히스토리의 형태를 결정한다. GitHub는 merge commit / squash /
rebase 세 방식을 제공하며, 실험으로 각각의 히스토리 형태를 직접
확인했다(참고 섹션).

업계 다수는 squash를 쓴다. 그 전제는 "브랜치 커밋은 어차피 지저분하므로,
main에는 PR 단위로 뭉갠 커밋 하나만 남기는 것이 낫다"이다. 그러나 이
프로젝트는 커밋을 의미 단위로 쪼개는 것 자체가 훈련 목표다(PROJECT.md 1조).
즉 이 프로젝트에서 브랜치의 커밋은 뭉갤 소음이 아니라 보존할 산출물이며,
squash의 전제가 성립하지 않는다.

## 결정

모든 PR을 merge commit으로 머지한다.  Ruleset의 Allowed merge methods를
Merge로만 제한하여 설정으로 강제한다.

## 결과

- **얻는 것**: 브랜치에서 다듬은 커밋들이 의미 단위 그대로 main에 보존된다.
  변경 이력을 커밋 단위로 추적할 수 있고, 문제 발생 시 git bisect / blame의
  분해능이 PR 단위가 아니라 커밋 단위가 된다. "브랜치를 따서 작업하고
  합쳤다"는 사건 구조도 그래프에 남는다.
- **잃는 것**: 히스토리 그래프의 시각적 복잡도. PR마다 레일 분기와 머지
  커밋이 하나씩 쌓인다.
- **감수 근거**: 기여자가 둘(maintainer, Claude Code)뿐이라 PR 유량이 적어
  복잡도 증가가 완만하다. PR 단위 조망이 필요할 때는
  `git log --first-parent`로 머지 커밋만 따라가면 된다. 또한 커밋 품질이
  훈련 목표대로 유지되는 한, 보존된 커밋은 소음이 아니라 정보다.

## 고려한 대안

1. **Squash and merge** — 업계 다수파. 커밋 위생을 개인 규율이 아니라
   시스템으로 해결하는 접근이나, 맥락에 쓴 대로 그 전제가 이 프로젝트에
   성립하지 않는다. 정성껏 쪼갠 커밋을 뭉개는 것은 훈련 산출물의 파괴다.
2. **Rebase and merge** — 커밋은 보존되지만 "한 PR이었다"는 경계 정보가
   사라지고, 머지 시 커밋이 재작성되어 해시가 바뀐다. 실험에서 직선
   히스토리라는 미학 외의 실익을 확인하지 못해 기각.
3. **이원화 (백엔드 merge / 프론트 squash)** — Claude Code의 커밋 품질을
   통제하기 어렵다는 우려에서 검토. 그러나 GitHub 설정으로는 강제할 수
   없어 머지하는 사람의 규율로만 유지되고, 규칙 복잡도 대비 이득이
   불분명하여 기각. 대신 프론트 커밋 규칙을 frontend/CLAUDE.md에 명시해
   같은 컨벤션을 따르게 하고, 미달 시 PR 리뷰에서 수정을 요구하는 것으로
   통제한다.
