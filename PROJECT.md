# PROJECT.md — 프로젝트 헌장

이 문서는 thirty-days 프로젝트의 목적, 구조, 역할, 규칙을 정의하는 최상위
문서다. 다른 모든 문서와 결정은 이 헌장의 목적에 복무한다. 헌장과 현실이
어긋나면 어느 쪽을 고칠지 결정하고, 헌장을 고칠 경우 그 이유를 기록한다.

---

## 1. 목적

이 프로젝트는 게시판을 만드는 프로젝트가 아니라 **운영하는 프로젝트**다.

3년차 백엔드 개발자로서 다음 질문에 측정된 답을 얻는 것이 목적이다:

> **"작은 시스템 하나를 요구사항 정의부터 배포·운영까지 혼자 완결할 수 있는가.
> 그리고 그 과정의 모든 결정을 자기 논리로 방어할 수 있는가."**

부차 목적:

- AI 도움 없이 가능한 영역과 AI에 위임해도 되는 영역의 경계를 실측한다
- 실무에서 겪기 어려운 경험(외부 공개, SLO 운영, 장애 대응, API 계약 협상)을
  통제된 환경에서 훈련한다
- 오픈소스 협업 관행(이슈, PR, 리뷰, 릴리스)을 실제로 운용해본다

**명시적 비(非)목적**: 실사용자 확보, 수익화, 기능 완성도 경쟁.
스코프 추가 제안은 이 비목적 목록과 대조하여 심사한다.

## 2. 프로젝트 구조

| 단계 | 기간 | 내용 | 완료 기준 |
|------|------|------|-----------|
| **0막 — 준비** | 1~2주 | Git 워크플로 리허설, 킥오프 문서,<br>요구사항/SLO 정의 | 요구사항 문서와 SLO 선언문 머지 |
| **1막 — 구축** | 4~6주 | Q&A 게시판 백엔드 구현 (AI 없이),<br>프론트엔드 병렬 개발 (Claude Code),<br>시뮬레이터 구축 | 로컬 환경에서 전 기능 동작,<br>시뮬레이터 트래픽 수용 |
| **간막 — 공개** | 1~2주 | 도메인, VPS, TLS, 배포 파이프라인 | 실 도메인에서 HTTPS 서비스 접근 가능,<br>무중단 배포 1회 성공 |
| **2막 — 운영** | 30일 | 공개 상태로 SLO 준수 운영,<br>시뮬레이터 상시 트래픽,<br>주간 이벤트 데이 | 30일 경과 + 최종 운영 보고서 작성 |

프로젝트의 본편은 2막이다. 1막의 완성도는 2막을 시작하기에 충분하면 된다.

## 3. 역할

| 역할 | 담당 | 권한과 책임 |
|------|------|------------|
| **Maintainer / Backend** | limecoding | 모든 결정의 최종 결정권자.<br>백엔드 구현, ADR 작성, 머지 권한 |
| **Frontend** | Claude Code | frontend/ 하위의 구현.<br>OpenAPI 계약 기반, PR로만 기여.<br>페르소나 정의: CLAUDE.md |
| **PL / Reviewer** | Claude (claude.ai) | 방향 제시, 스코프 감시, 리뷰, 페이스 관리.<br>**결정권 없음.** 결정을 대신하지 않는다 |
| **Users** | LLM 페르소나 에이전트 +<br>부하 생성기 (simulator/) | 콘텐츠 생성과 트래픽.<br>2막의 상시 배경 부하 담당 |

### PL의 행동 규약

- 지킨다: 스코프 감시, 결정 근거 요구("이거 ADR 감인데요"), 1막 규칙 감시,
  일정보다 페이스 우선
- 하지 않는다: 결정 대행, 1막 중 백엔드 코드 작성. 막힐 때는 답이 아니라
  "어디를 봐야 하는지"까지만 좁힌다

### PL 세션 프로토콜

- 세션 시작: 리포 URL 공유 → PL이 STATUS.md와 관련 문서를 읽고 싱크
- 세션 종료: 결정 사항을 STATUS.md에 반영하여 커밋 (maintainer 담당).
  **회의록 반영까지가 세션 종료의 정의다**
- PL은 세션 간 기억이 보장되지 않는다. 리포지토리가 유일한 진실이다

## 4. AI 사용 규칙 — 계층 모델

이 프로젝트의 핵심 실험 설계. 영역별로 AI와의 관계를 다르게 설정한다.

| 계층 | 영역 | 규칙 |
|------|------|------|
| **1층 — AI 금지** | 1막의 백엔드 전체<br>(도메인 로직, DB, 테스트) | 공식 문서와 레퍼런스만 사용.<br>AI에게 코드 작성·디버깅을 요청하지 않는다 |
| **2층 — AI 협업** | 인프라, 배포, 모니터링,<br>시뮬레이터 | AI 사용 가능. 단 모든 주요 결정은<br>ADR로 남기고 근거를 자기 언어로 쓴다 |
| **3층 — AI 위임** | 프론트엔드 | Claude Code에 위임. 단 PR 리뷰로 통제하고<br>API 계약(openapi.json)으로 경계를 강제한다 |

1층 규칙의 예외: 막힌 상태로 자력 시도가 상당 시간 지속되면, PL에게
"어디를 봐야 하는지"를 물을 수 있다. 답을 받는 것이 아니라 탐색 방향을
좁히는 것까지만 허용된다.

## 5. 운영 원칙

- **문서화되지 않은 합의는 존재하지 않는 것으로 간주한다.**
  프로젝트의 진실은 이 리포지토리에만 있다.
- **모든 주요 결정은 ADR로 남긴다.** 기준: 바꾸기 비싸거나, 나중에
  "왜?"라고 물을 것 같은 결정. 절차는 CONTRIBUTING.md의 PR 등급제를 따른다.
- **장애 발생 시, 고치기 전에 가설을 먼저 기록한다.** 가설이 틀려도
  기록을 남긴다. 디버깅 정확도의 측정이 목적이다.
- **회고는 마일스톤 단위로 한다.** 첫 회고(v0.1)에서 프로세스 등급제의
  요율을 재조정한다. 회고 질문: "프로세스 때문에 미룬 작업이 있었나?"
- **페이스가 일정에 우선한다.** 기간 추정치는 목표가 아니라 참고치다.
  이 프로젝트의 실패 조건은 지연이 아니라 중단이다.
